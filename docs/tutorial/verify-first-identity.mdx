This tutorial takes you from a Sandbox bearer token to your first completed identity verification. You call `POST /v3/verify` for **Human Assurance**, read a passing result, then trigger a failing result and read the reasons. By the end you can interpret the fields that decide whether to **allow**, **step up**, or **block** a user.

This tutorial uses **Human Assurance**, a server-only flow that needs no client SDK, so you can complete every step with cURL. For the wider product set, see [What's next](#whats-next).

:::info
Human Assurance is available only in the **United States**.
:::

## What you learn

- Exchange Sandbox credentials for a bearer token
- Send a verification request to `POST /v3/verify` with a Sandbox test user
- Read `success`, `identity.assuranceLevel`, and `evaluation` to understand a passing result
- Trigger a failing result and map its failure reasons

## Prerequisites

- **Human Assurance Sandbox credentials**: An OAuth **client ID** and **client secret** for a Human Assurance project in the Developer Portal.
- **A bearer token**: This guide recaps the token step. For the full walkthrough, complete [Get started with Prove API authentication](./access-api-keys) first.
- **An HTTP client**: A terminal with **cURL**, or a client such as Postman, to run the requests.

### 1. Get a Sandbox bearer token

Prove Platform APIs use **OAuth 2.0 client credentials**. Exchange your Sandbox **client ID** and **client secret** for a bearer token. Replace the placeholders with your values from the **Credentials** tab of your Human Assurance project.

```bash title="Request a bearer token (US Sandbox)"
curl -X POST https://platform.uat.proveapis.com/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials&client_id=YOUR_CLIENT_ID&client_secret=YOUR_CLIENT_SECRET"
```

Copy the **`access_token`** from the response. Tokens are valid for **1 hour**. Request a new one when it expires.

:::tip[Checkpoint]
You receive **HTTP 200** with `"token_type": "Bearer"` and a non-empty **`access_token`**.
:::

### 2. Verify a known-good identity

Send your first verification. Use the Sandbox test user **Ewen Brimilcome**, whose phone number always passes. Replace `YOUR_ACCESS_TOKEN` with the token from Step 1.

For Human Assurance, set `verificationType` to `humanAssurance` and include `phoneNumber`. The `clientRequestId` is your own identifier for the request.

```bash title="Verify Ewen Brimilcome (passes)"
curl -X POST "https://platform.uat.proveapis.com/v3/verify" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "verificationType": "humanAssurance",
    "phoneNumber": "2001004051",
    "clientRequestId": "tutorial-001"
  }'
```

:::tip[Checkpoint]
You receive **HTTP 200** with `"success": "true"`. An **HTTP 401** means the bearer token is missing, wrong, or expired. Repeat Step 1.
:::

### 3. Read the verification result

The response carries the signals you act on. Here is the passing result for Ewen.

```json title="Success response"
{
    "success": "true",
    "correlationId": "39614b5f-b123-46bb-b36c-43eb930aa344",
    "clientRequestId": "tutorial-001",
    "phoneNumber": "2001004051",
    "proveId": "7fe34f25-4c71-4589-8330-0bf265374528",
    "identity": {
        "assuranceLevel": "AL1"
    },
    "evaluation": {
        "authentication": {
            "result": "pass"
        },
        "risk": {
            "result": "pass"
        }
    },
    "isEnrolled": true
}
```

Read these fields in order:

- **`success`**: The top-level pass or fail. Branch your UX and backend logic on this value.
- **`identity.assuranceLevel`**: Your confidence signal for human-versus-automation policy.
- **`evaluation`**: The authentication and risk outcomes. Interpret them with the Global Fraud Policy. For Human Assurance, weigh both `authentication` and `risk` when you tune thresholds.
- **`proveId`**: A stable identifier for the verified identity. Persist it when you need support, auditing, or linking to other Prove flows.

:::tip[Checkpoint]
You can point to the field that holds the pass or fail decision and the field that holds the assurance level.
:::

### 4. See a failed verification

A real integration handles failures, so trigger one now. Use the Sandbox test user **Hilary Kumaar**, whose phone number always fails.

```bash title="Verify Hilary Kumaar (fails)"
curl -X POST "https://platform.uat.proveapis.com/v3/verify" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "verificationType": "humanAssurance",
    "phoneNumber": "2001004052",
    "clientRequestId": "tutorial-002"
  }'
```

The failing response carries `failureReasons` codes you map to a policy action.

```json title="Failure response"
{
    "success": "false",
    "correlationId": "afdbcc66-d67c-4523-b068-2c749f7cad27",
    "clientRequestId": "tutorial-002",
    "phoneNumber": "2001004052",
    "identity": {
        "assuranceLevel": "AL-1"
    },
    "evaluation": {
        "authentication": {
            "failureReasons": {
                "9176": "The phone number and identity is strongly associated with a fraud vector."
            },
            "result": "fail"
        },
        "risk": {
            "failureReasons": {
                "9304": "Suspicious large amount of recent activity across different identity attributes."
            },
            "result": "fail"
        }
    },
    "isEnrolled": false
}
```

Read `evaluation.authentication.failureReasons` and `evaluation.risk.failureReasons`, then map each code with the Global Fraud Policy. A low `assuranceLevel` such as `AL-1` signals a failed human-presence read.

:::tip[Checkpoint]
You receive **HTTP 200** with `"success": "false"` and at least one code under `failureReasons`.
:::

## Summary

You ran a complete identity verification end to end. You exchanged Sandbox credentials for a bearer token, called `POST /v3/verify` for Human Assurance, read a passing result, and mapped the reasons behind a failing result. You now know which fields drive an allow, step-up, or block decision.

## What's next

- **Move into your codebase**: Install a Prove server SDK and wire in your Sandbox credentials with [Set up Prove SDKs in your project](../how-to/dev-environment-sdk-setup).
- **Build the production flow**: Use Human Assurance Verify for the full request schema, optional identifiers, and a troubleshooting reference.
- **Keep identities current**: Enroll passing identities in continuous monitoring with Human Assurance Manage.
- **Explore other products**: Each solution under **Welcome** and **Know** has its own Authenticate and Verify guides, such as Pre-Fill for Consumers, Account Opening, and Unified Authentication.
