This tutorial takes you from a Sandbox bearer token to your first completed identity verification. You call `POST /v3/verify` for **Human Assurance**, read a passing result, then trigger a failing result and compare the two. By the end you can interpret the fields that decide whether to **allow**, **step up**, or **block** a user.

This tutorial uses **Human Assurance**, a server-only flow that needs no client SDK, so you can complete every step with cURL. For the wider product set, see [What's next](#whats-next).

:::info
Human Assurance is available only in the **United States**.
:::

## What you learn

- Exchange Sandbox credentials for a bearer token
- Send a verification request to `POST /v3/verify` with a Sandbox test user
- Read `identity.assuranceLevel`, `proveId`, and `isEnrolled` to understand a passing result
- Trigger a failing result and recognize it by its assurance level

## Prerequisites

- **Human Assurance Sandbox credentials**: An OAuth **client ID** and **client secret** for a Human Assurance project in the Developer Portal.
- **A bearer token**: This guide recaps the token step. For the full walkthrough, complete [Get started with Prove API authentication](https://papadewald86.github.io/portfolio/docs/tutorial/access-api-keys) first.
- **An HTTP client**: A terminal with **cURL**, or a client such as Postman, to run the requests.

### 1. Get a Sandbox bearer token

Prove Platform APIs use **OAuth 2.0 client credentials**. Exchange your Sandbox **client ID** and **client secret** for a bearer token. Replace the placeholders with your values from the **Credentials** tab of your Human Assurance project.

```bash title="Request a bearer token (US Sandbox)"
curl -X POST https://platform.uat.proveapis.com/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials&client_id=YOUR_CLIENT_ID&client_secret=YOUR_CLIENT_SECRET"
```

Copy the **`access_token`** from the response. Tokens are valid for **1 hour**. Request a new one when it expires.

:::tip[Checkpoint]
You receive **HTTP 200** with `"token_type": "Bearer"` and a non-empty **`access_token`**.
:::

### 2. Verify a known-good identity

Send your first verification. Use the Sandbox test user **Ewen Brimilcome**, whose phone number always passes. Replace `YOUR_ACCESS_TOKEN` with the token from Step 1.

For Human Assurance, set `verificationType` to `humanAssurance` and include `phoneNumber`. The `clientRequestId` is your own identifier for the request.

```bash title="Verify Ewen Brimilcome (passes)"
curl -X POST "https://platform.uat.proveapis.com/v3/verify" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "verificationType": "humanAssurance",
    "phoneNumber": "2001004051",
    "clientRequestId": "tutorial-001"
  }'
```

:::tip[Checkpoint]
You receive **HTTP 200** with `"assuranceLevel": "AL1"` and a non-empty **`proveId`**.
:::

{/* step { "httpRequest": { "url": "https://platform.uat.proveapis.com/v3/verify", "method": "POST", "request": { "headers": { "Authorization": "Bearer $HA_ACCESS_TOKEN", "Content-Type": "application/json" }, "body": { "verificationType": "humanAssurance", "phoneNumber": "2001004051", "clientRequestId": "tutorial-001" } }, "response": { "body": { "identity": { "assuranceLevel": "AL1" } }, "required": ["proveId", "isEnrolled"] }, "statusCodes": [200] } } */}

### 3. Read the verification result

The response carries the signals you act on. Here is the passing result for Ewen.

```json title="Success response"
{
    "correlationId": "95ee4cef-ec2f-47b6-a3d0-6a6dd41f966b",
    "clientRequestId": "tutorial-001",
    "phoneNumber": "2001004051",
    "proveId": "7fe34f25-4c71-4589-8330-0bf265374528",
    "provePhoneAlias": "XZM1NPS5S24E775YJHG6AMRG42X292ZDZ7J527WD5YKUDFYU5D4BMR94Z0P2BAJQUQHCD77LCE92T6E195S7FS3FB2PLCEUCBJHFAK70G8SY1JTC3RAS07S4HUR64G66",
    "identity": {
        "assuranceLevel": "AL1"
    },
    "isEnrolled": true
}
```

Read these fields in order:

- **`identity.assuranceLevel`**: The verification outcome and your confidence signal for human-versus-automation policy. **`AL1`** is a passing read — branch your UX and backend logic on this value.
- **`proveId`**: A stable identifier for the verified identity, returned only when verification passes. Persist it when you need support, auditing, or linking to other Prove flows.
- **`isEnrolled`**: Whether the identity was enrolled for continuous monitoring under the Global Fraud Policy.
- **`provePhoneAlias`**: An opaque, stable alias for the verified phone number.

:::tip[Checkpoint]
You can point to the field that holds the verification outcome and the field you persist for later Prove flows.
:::

### 4. See a failed verification

A real integration handles failures, so trigger one now. Use the Sandbox test user **Hilary Kumaar**, whose phone number always fails.

```bash title="Verify Hilary Kumaar (fails)"
curl -X POST "https://platform.uat.proveapis.com/v3/verify" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "verificationType": "humanAssurance",
    "phoneNumber": "2001004052",
    "clientRequestId": "tutorial-002"
  }'
```

The failing response returns a low assurance level — and no `proveId` to act on.

```json title="Failure response"
{
    "correlationId": "2df1ad4b-4185-416a-9650-cbdd5056c00a",
    "clientRequestId": "tutorial-002",
    "phoneNumber": "2001004052",
    "identity": {
        "assuranceLevel": "AL-1"
    },
    "isEnrolled": false
}
```

Compare it to the passing response: a low `assuranceLevel` such as `AL-1` signals a failed human-presence read, no `proveId` or `provePhoneAlias` is issued, and `isEnrolled` is `false`. Branch on the assurance level and the absence of `proveId`, and route the user to your fallback policy — block, or step up to another verification method.

:::tip[Checkpoint]
You receive **HTTP 200** with `"assuranceLevel": "AL-1"` and no **`proveId`** in the response.
:::

## Summary

You ran a complete identity verification end to end. You exchanged Sandbox credentials for a bearer token, called `POST /v3/verify` for Human Assurance, read a passing result, and recognized a failing one by its assurance level. You now know which fields drive an allow, step-up, or block decision.
