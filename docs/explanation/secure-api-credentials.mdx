To obtain OAuth client credentials and bearer tokens, see [Get started with Prove API authentication](https://papadewald86.github.io/portfolio/docs/tutorial/access-api-keys).

Secret API keys and client credentials behave like account passwords. If attackers expose them, they can abuse your integration and data.

You're responsible for protecting Prove credentials in your environments.

## Key concepts

- **Client ID:** Public OAuth client identifier used with Prove token endpoints. Treat it as sensitive configuration, not as a secret you embed in client-side apps.
- **Client secret:** Confidential value used on your **server** to obtain access tokens. Store only in a secrets manager or KMS; never commit it or ship it in mobile or browser code.
- **Bearer token:** Short-lived credential sent on Platform API calls after a successful token request. Issue and cache on the server; don't treat it as a long-lived secret in front-end or mobile binaries.

## Protect against compromised credentials

- **Use a key management system (KMS) or secrets manager for production secrets.** When you create or rotate a production client secret, store it only in a system designed for secrets. Avoid leaving copies in local files or shared drives.
- **Limit who can create, read, or rotate keys.** Define ownership, use least privilege, and review access periodically.
- **Don't share secrets in email, chat, or support tickets.** Use approved secret-handling channels only.
- **Don't commit secrets to source control.** Public repositories are actively scanned; private repos still leak through clones and tooling. Use environment variables or your CI/CD secret store, never literals in code.
- **Don't embed client secrets or long-lived tokens in client-side apps.** Mobile binaries and front-end JavaScript can be reverse-engineered. Treat anything shipped to the device as public.
- **Check API usage.** Review logs for unusual patterns and ensure Sandbox credentials aren't used against Production endpoints.
- **Train your team and document your process.** Keep internal processes current for onboarding, incident response, and rotation.
- **Rotate credentials on a schedule and after suspected exposure.** Rotating client secrets and invalidating old values limits blast radius if a secret leaks without your knowledge.
- **Avoid logging credentials.** Configure logging and error handling so client secrets, bearer tokens, and full request/response bodies containing them are never written to application or third-party logs.

For HTTP errors when calling Platform APIs, see Errors and status codes.
